# work-tracker

## overview

- tracks freelance hours worked for various clients, along with the position worked and pay rate

- generates invoices for pay-by-invoice gigs and emails them to the client.

- extracts information about possible past or upcoming jobs from emails sent from particular addresses or domains and prompts user to input data for those jobs

- aggregates and displays data for money made, hours worked, etc by date, position, tag, and/or client for chosen periods

## architecture

firebase backend.

should be usable as a webpage, or an iPhone app. please make some suggestions for how to approach this. it will be single-user for now. may grow to multi-user in the future.

## ux design

design and feel inspired by vscode and qlab. minimal ornamentation. use of icons to impart information is encouraged, but only if it doesn't degrade understandability. toolstips to provide context. similar controls or workflows should share ux elements rather than reinventing new ways of doing similar things. design refactoring to maintain a common approach is encouraged.

## coding

coding will be done by the ai, directed by the app owner.
Code should be self-describing as much as possible, and where it becomes complicated or difficult to reason about should be refactored immediately.
code should always be kept DRY and SOLID with full test coverage at every point. tests should cover unit, integration and end-to-end.
tests should cover happy path, error, bad input and unexpected failure mid process as well as any other specific situations relevant to the code under test.

## ai infrastructure

the ai will be used to generate code, tests, documentation and design. the ai will be used to refactor code and design as needed. the ai will be used to generate commit messages and pull request descriptions. the ai will be used to generate user-facing content such as emails and invoices. the ai will be used to generate data for testing. the ai will be used to generate prompts, instructions, agents, subagents, instructions, tools or any other resources for itself to use in generating code, tests, documentation and design as necessary.

- always prioritize maintainability, readability and scalability in your code, tests, documentation and design. always strive to keep design, code and documentation DRY and SOLID. this will help ensure that the app can be easily updated and improved in the future as needed.
- keep in mind that the app is intended to be used by a single user for now, but may grow to multi-user in the future. design and code accordingly to allow for easy expansion to multi-user functionality if needed.
- always keep the user experience in mind when generating code, tests, documentation and design. strive to create an intuitive and user-friendly interface that makes it easy for the user to track their hours, generate invoices and view their data.
